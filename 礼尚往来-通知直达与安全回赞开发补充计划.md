## 1. 最终决策

### 1.1 评论回复的主链路

评论继续以**创作者评论管理页**作为数据来源和回复入口：

```text
评论管理页扫描评论
→ 写入 interaction_events
→ comments:plan 生成回复计划
→ 用户填写 replyText、设置 approved
→ comments:reply 根据计划回到原评论位置
→ 页面确认回复结果
→ actions 记录动作
```

不要将评论回复改成依赖通知面板逐条跳转。评论管理页更容易拿到完整的评论正文、所属作品、是否已回复和原地回复入口。

### 1.2 点赞回访的主链路

点赞回访应选择**通知条目直达真实用户主页**，而不是“根据昵称重新搜索用户”：

```text
通知面板发现点赞事件
→ 从该通知条目提取或补全用户主页入口
→ 写入事件记录
→ 仅对 friend / mutual 生成计划
→ 根据已保存的主页入口定位目标视频
→ 人工审批
→ 执行阶段只依据计划目标点赞
```

### 1.3 用户名搜索的定位

按昵称搜索只能是**人工兜底工具**，不能成为自动计划或自动执行主路径。

```text
通知中拿不到主页入口
→ 事件标记 blocked
→ 用户人工选择是否启用搜索辅助
→ 用户人工确认候选主页
→ 保存稳定主页 URL / Key
→ 重新生成计划
```

禁止：

```text
通知只提取昵称
→ 程序自动搜索昵称
→ 自动选择第一个搜索结果
→ 自动点赞
```

原因：昵称不唯一、可修改、搜索排序不稳定，极易给错误用户回赞。

# 3. 紧急风险审查：必须先封住真实点赞入口

以下问题为 P0 级阻塞项。未修复前，不继续增强回赞功能，不进行真实账号回赞测试。

---

## 3.1 P0-1：`likes:reciprocate` 不读取计划，绕过 `approved`

### 当前现象

`src/cli/plan-likes.mjs` 已经可以生成包含以下信息的计划条目：

```js
{
  eventId,
  actorName,
  relation,
  actorProfileUrl,
  targetVideoUrl,
  targetVideoTitle,
  alreadyLiked,
  approved: false,
  status: 'planned'
}
```

但是，`src/cli/execute-reciprocal-likes.mjs` 当前没有读取 `--plan <文件>`，而是重新：

```text
打开通知面板
→ extractNotifications()
→ 过滤点赞通知
→ processOneLike()
→ 找主页和视频
→ 点赞
```

### 风险

- 用户编辑计划文件和设置 `approved: true` 没有作用；
- 执行程序不受人工审批控制；
- 计划阶段确认的主页/视频目标不会被执行阶段复用；
- 执行阶段可能操作到另一条同名或新出现的通知。

### 必须改为

```bash
npm run likes:reciprocate -- --plan data/plans/likes-plan-xxx.json --dry-run
npm run likes:reciprocate -- --plan data/plans/likes-plan-xxx.json --execute --max-items 1
```

执行程序必须：

1. 读取计划文件；
2. 只读取 `approved === true` 的条目；
3. 只按计划内的 `eventId`、`actorProfileUrl`、`targetVideoUrl` 执行；
4. 不重新扫描通知列表来决定执行对象。

未提供 `--plan` 时，不得进入真实执行流程。

## 3.2 P0-2：当前 dry-run 仍可能调用真实 `clickLike()`

### 当前现象

执行脚本会打印模式：

```text
execute / dry-run
```

但在处理单条点赞的函数中，当前流程仍直接进入：

```js
const likeResult = await clickLike(page);
```

没有在调用真实点击前判断：

```js
run.options.execute === true
```

### 风险

用户以为自己在“只检查、不操作”，实际可能已经给视频点赞。

### 双重保护要求

必须在两层同时增加保护。

#### 第一层：Workflow / CLI 层

```js
if (run.options.dryRun || !run.options.execute) {
  return {
    status: 'dry-run-ready',
    reason: '目标已定位，dry-run 不执行点赞'
  };
}
```

#### 第二层：Adapter 层

修改 `src/adapters/video-page.mjs`：

```js
export async function clickLike(page, { execute = false } = {}) {
  if (!execute) {
    return blocking(
      RESULT_CODES.ACTION_NOT_APPROVED,
      '非 execute 模式，拒绝真实点赞操作',
      { recoverable: false }
    );
  }

  // 仅 execute=true 才进入真实 DOM click
}
```

任何未来误调用 `clickLike(page)`，都不得默认点击。

## 3.3 P0-3：当前执行阶段按昵称反查事件，无法可靠绑定本人

### 当前现象

执行脚本使用如下类型的查询找事件：

```sql
SELECT id
FROM interaction_events
WHERE actor_name = ?
  AND event_type = 'like'
  AND status = 'new'
ORDER BY created_at DESC
LIMIT 1
```

### 风险

- 同名用户会串错；
- 同一用户多次点赞无法区分；
- 计划阶段的事件可能已改为 `planned`，执行阶段只查 `new`，反而找不到；
- 执行脚本重新扫描通知后获得的对象，未必是计划时确认的对象；
- 操作审计无法证明给谁回赞。

### 必须改为

执行阶段以计划中的唯一信息为输入：

```js
item.eventId
item.eventFingerprint
item.actorProfileUrl
item.actorProfileKey
item.targetVideoUrl
```

数据库校验必须按 `eventId`：

```sql
SELECT *
FROM interaction_events
WHERE id = ?
  AND event_type = 'like'
  AND status IN ('planned', 'approved')
```

禁止执行脚本以昵称作为事件匹配主键。

## 3.4 P0-4：点赞状态未知时，当前仍可能进入点击

### 当前现象

在计划脚本中，目前只有“明确已点赞”时跳过：

```js
if (likeResult.ok && likeResult.data.alreadyLiked) {
  // skipped
}
```

如果 `checkLikeState()` 返回失败或未知，代码仍可能将该任务视为候选。

执行脚本也存在同类问题：只拦截“明确已点赞”，但状态识别失败时仍可继续进入 `clickLike()`。

### 风险

点赞是切换动作：

```text
未赞状态点击 → 点赞
已赞状态点击 → 取消点赞
```

当状态未知时点击，可能把之前已经点过的赞取消掉。

### 强制规则

只有满足以下状态才允许真实点击：

```js
stateResult.ok === true
stateResult.data.alreadyLiked === false
stateResult.data.confidence === 'confirmed'
```

任何无法明确识别的情况：

```text
status = blocked
code = LIKE_STATE_UNKNOWN
不得点击
```

如果当前 `checkLikeState()` 尚无 `confidence` 字段，应新增：

```js
{
  alreadyLiked: false,
  confidence: 'confirmed',
  signal: '明确识别到的页面信号'
}
```

# 4. 身份绑定问题：通知直达必须变成“稳定事件入口”

当前选择“点击通知头像进入主页”的方向正确，但代码仍然过度依赖昵称重新定位通知条目。

---

## 4.1 当前通知提取数据不足

`extractNotifications()` 当前主要返回：

```js
{
  username,
  relation,
  eventType,
  action,
  content,
  timeText
}
```

当前没有稳定保存：

```js
actorProfileUrl
actorProfileKey
rawText
notificationItemKey
```

`scan-interactions.mjs` 写库时也没有将主页 URL / Key 写入互动事件。

### 后果

后续计划阶段只能重新在面板里用昵称找头像：

```js
clickLikeProfileLink(page, event.actor_name)
```

这仍然不能证明点击的是原始点赞者本人。

---

## 4.2 通知扫描应升级为“事件 + 用户入口”采集

### 推荐数据结构

每条点赞通知扫描结果尽可能返回：

```js
{
  eventType: 'like',
  username: '用户昵称',
  relation: 'friend',
  action: '赞了你的作品',
  timeText: '2小时前',
  actorProfileUrl: 'https://www.douyin.com/user/MS4wLjAB...',
  actorProfileKey: 'MS4wLjAB...',
  rawText: '整条通知原始文本',
  notificationItemKey: '通知项指纹'
}
```

并写入数据库已有字段：

```text
actor_profile_url
actor_profile_key
raw_payload_json
fingerprint
```

### 指纹要求

点赞事件的指纹不能只依赖昵称。建议构成为：

```text
eventType
+ actorProfileKey 或 actorProfileUrl
+ action
+ timeText
+ rawText 标准化摘要
```

---

## 4.3 若通知 DOM 中拿不到 href，允许的降级方案

有些站点头像点击后跳转，但 DOM 中不直接暴露用户 URL。此时允许做一个独立的“入口解析”步骤：

```text
扫描出点赞事件
→ 在 dry-run / resolve 阶段精准定位该通知条目
→ 点击头像，仅发生页面导航
→ 读取进入后的主页 URL / profileKey
→ 更新 interaction_events
→ 返回通知页处理下一条
```

### 该降级方案的边界

- 点击头像进入主页不是点赞动作，可以在 dry-run 解析阶段允许；
- 但必须通过整条通知指纹精准定位，不能只用昵称 `includes()`；
- 成功解析到主页入口后必须保存，执行阶段不应再次依赖通知面板；
- 无法解析主页入口的事件一律 `blocked`；
- 不得自动转为昵称搜索。

---

## 4.4 `clickLikeProfileLink()` 必须去掉昵称模糊匹配风险

当前逻辑以昵称片段进行包含匹配，存在如下误点情形：

```text
昵称相同
昵称互为子串
同一用户存在多条不同通知
点赞通知与评论通知混杂
```

改造方向：

### 首选方案

扫描阶段直接提取该条通知里的用户 `<a href>`，不再需要 `clickLikeProfileLink()` 作为计划主流程。

### 可接受方案

若必须点击通知条目解析 URL，函数参数不得只有 `username`，必须接收完整事件描述：

```js
clickNotificationActorProfile(page, {
  username,
  relation,
  action,
  timeText,
  fingerprint,
  rawText
})
```

只有精确匹配目标通知条目时才允许导航。

# 5. 计划生成阶段应只确定目标，不产生真实互动

## 5.1 当前 `findLatestNonPinnedVideo()` 有导航副作用

当前函数命名是“查找最新非置顶视频”，但内部已经执行：

```js
first.click();
```

随后调用方又显式执行：

```js
navigateToVideo(page, videoUrl)
```

这会让计划生成阶段出现隐式页面变化，不利于调试与审计。

### 必须拆分

```js
findLatestNonPinnedVideo(page)   // 只读取候选，不点击、不导航
navigateToVideo(page, videoUrl)  // 由 workflow 明确调用
```

### 返回结果应包含

```js
{
  ok: true,
  data: {
    videoUrl,
    videoId,
    targetRule: 'latest_non_pinned_video',
    candidateCount,
    isPinned: false,
    evidence: {}
  }
}
```

---

## 5.2 `likes:plan` 的正确职责

`likes:plan` 应当负责：

```text
读取已绑定主页入口的点赞事件
→ 校验好友关系
→ 打开保存的 actorProfileUrl
→ 验证身份
→ 读取候选目标视频
→ 显式打开目标视频
→ 明确判断当前未点赞
→ 生成 approved=false 的计划
```

不得负责：

- 真实点击点赞；
- 自动批准；
- 以昵称重新搜索用户；
- 状态未知时将任务纳入可执行计划。

## 6.1 阶段一：扫描事件

命令示例：

```bash
npm run interactions:scan -- --type like --dry-run --debug --pause-on-error
```

输出事件应包含：

```json
{
  "eventId": 12,
  "actorName": "好友昵称",
  "relation": "friend",
  "actorProfileUrl": "https://www.douyin.com/user/...",
  "actorProfileKey": "MS4wLjAB...",
  "eventFingerprint": "...",
  "status": "new"
}
```

没有稳定主页入口的事件：

```text
可以保存以便分析
但不得进入可执行回赞计划
```

---

## 6.2 阶段二：生成计划

命令示例：

```bash
npm run likes:plan -- --dry-run --max-items 10 --keep-open-on-error
```

计划文件：

```json
{
  "planType": "reciprocal_like",
  "mode": "manual",
  "planId": 3,
  "createdAt": "2026-05-29T00:00:00.000Z",
  "items": [
    {
      "eventId": 12,
      "eventFingerprint": "...",
      "actorName": "好友昵称",
      "actorProfileUrl": "https://www.douyin.com/user/...",
      "actorProfileKey": "MS4wLjAB...",
      "relation": "friend",
      "targetRule": "latest_non_pinned_video",
      "targetVideoUrl": "https://www.douyin.com/video/...",
      "targetVideoId": "...",
      "targetVideoTitle": "视频标题",
      "likeState": "confirmed_not_liked",
      "evidencePath": "data/runs/<run-id>/evidence/...",
      "approved": false,
      "status": "planned",
      "reason": ""
    }
  ]
}
```

### 只有以下条件全满足，条目才可为 `planned`

```js
relation === 'friend' || relation === 'mutual'
actorProfileUrl 非空
actorProfileKey 非空或具备经过验证的稳定身份校验方式
targetVideoUrl 非空
likeState === 'confirmed_not_liked'
数据库中不存在 targetVideoUrl 的 succeeded like_work 动作
```

---

## 6.3 阶段三：人工审批

用户检查计划中的：

- 用户昵称；
- 关系；
- 用户主页；
- 目标视频标题/URL；
- 截图证据；
- 当前状态是否未点赞。

确认后，仅对要执行的一条设置：

```json
"approved": true
```

---

## 6.4 阶段四：dry-run 执行预览

命令：

```bash
npm run likes:reciprocate -- --plan data/plans/likes-plan-xxx.json --dry-run --max-items 1 --keep-open
```

允许行为：

```text
读取计划
→ 读取 approved 条目
→ 打开 targetVideoUrl
→ 校验页面与目标
→ 重新检查点赞状态
→ 保存截图
→ 输出“可执行 / 已点赞 / 状态未知”
```

禁止行为：

```text
clickLike()
写 succeeded
修改任何真实点赞状态
```

---

## 6.5 阶段五：单条真实执行

命令：

```bash
npm run likes:reciprocate -- --plan data/plans/likes-plan-xxx.json --execute --max-items 1 --keep-open-on-error --pause-on-error
```

严格校验顺序：

```text
1. 必须有 --plan
2. 计划文件必须存在且 planType=reciprocal_like
3. 只处理 approved=true 条目
4. eventId 必须存在
5. DB 中事件必须存在且 event_type=like
6. 事件状态必须为 planned/approved
7. relation 必须为 friend 或 mutual
8. actorProfileUrl 和 targetVideoUrl 必须存在
9. targetVideoUrl 不得已有 succeeded 点赞记录
10. 本轮执行数量未达到 max-items
11. 打开目标视频成功
12. 页面仍可确认是计划中的目标作品
13. 当前点赞状态明确为未赞
14. 当前模式明确为 execute
15. 点击点赞
16. 页面确认已赞
17. 保存证据并写 actions
18. 更新 event 为 succeeded
```

### 执行阶段禁止重新决定目标

如果计划目标失效，应：

```text
blocked
重新生成计划
```

不得在执行阶段重新：

- 扫描通知挑另一个用户；
- 按昵称搜索用户；
- 进入主页重新选另一个视频；
- 自动替换目标 URL。

# 7. 新增或调整的错误码建议

在已有错误码基础上，点赞链路建议至少完善以下语义：

| 错误码 | 使用场景 | 动作 |
|---|---|---|
| `PLAN_REQUIRED` | 执行命令缺少 `--plan` | 拒绝执行 |
| `PLAN_TYPE_MISMATCH` | 计划类型不是 `reciprocal_like` | 拒绝执行 |
| `ACTION_NOT_APPROVED` | 条目没有审批或 clickLike 非 execute 调用 | 拒绝点击 |
| `ACTOR_PROFILE_LINK_NOT_FOUND` | 通知无法绑定主页入口 | 阻塞 |
| `ACTOR_IDENTITY_UNCONFIRMED` | 主页不能确认与事件用户一致 | 阻塞 |
| `RELATION_UNKNOWN` | 无法确认好友/互关 | 跳过或阻塞 |
| `LATEST_VIDEO_NOT_FOUND` | 无明确目标作品 | 阻塞 |
| `LIKE_STATE_UNKNOWN` | 无法判断当前是否已点赞 | 阻塞，禁止点击 |
| `ALREADY_LIKED` | 当前明确已点赞 | 安全跳过 |
| `DUPLICATE_ACTION` | 已成功处理相同目标视频 | 跳过 |
| `TARGET_CHANGED` | 执行页与计划目标不一致 | 阻塞 |
| `RATE_LIMITED` | 验证码、频控或风控 | 停止整轮 |

# 8. 现有文件的具体修改建议

## 8.1 `src/cli/execute-reciprocal-likes.mjs`

### 当前定位

高风险，需要首先修改。

### 修改要求

- 增加 `--plan` 参数；
- 删除“执行时重新 `extractNotifications()` 决定对象”的主链路；
