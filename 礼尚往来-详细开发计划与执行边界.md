## 0. 本版修订说明

本文件是《礼尚往来-详细开发计划与执行边界.md》的新版替代文档。

旧版文档的核心安全原则仍然有效：

- 默认只观察和定位，不默认执行真实互动；
- 真实动作必须审批、可验证、可追溯、可停止；
- 页面状态不明确时必须阻断；
- 单轮真实执行默认最多 1 条；
- 风控、验证码、登录异常出现时立即停止。

本版主要修订以下过期内容：

| 旧版内容 | 新版处理 |

| 将项目主要定位为本地 CLI 互动工具 | 改为 **OpenClaw 创作者互动助手 Skill 的执行引擎** |

| 将单条真实回赞列为正常后续阶段 | 改为：非 MVP 的实验验证能力，必须受实验开关保护 |
| 只描述 CLI 开发任务 | 增加 `SKILL.md`、结构化输出、Skill 调用契约与审批状态机 |

本文件可以直接覆盖仓库根目录旧版同名文档。

# 1. 项目最终定位

## 1.1 产品身份

“礼尚往来”不是自动互赞脚本，不是刷互动增长工具，也不是无人值守运营机器人。

本项目应定位为：

> **面向抖音个人创作者的 OpenClaw 互动助手 Skill 执行引擎。**

它帮助用户：

1. 发现新增评论和互动通知；
2. 整理还未处理的真实互动；
3. 生成评论回复建议；
4. 生成好友/互关回访候选预览；
5. 在用户明确确认后，安全执行单条评论回复；
6. 在未来满足严格身份核验条件后，实验性验证单条回访点赞。

## 1.2 Skill 与执行引擎的关系

项目必须拆成两层理解：

```text
用户
  ↓ 自然语言
OpenClaw Agent + 礼尚往来 SKILL.md
  ├── 理解用户意图
  ├── 决定调用哪些只读/审批/执行动作
  ├── 向用户展示候选与风险
  ├── 收集明确审批
  └── 限制真实动作范围
        ↓ 调用本地命令
Node.js + Playwright + SQLite 执行引擎
  ├── 浏览器登录态复用
  ├── 页面读取与元素定位
  ├── 事件入库与去重
  ├── dry-run 定位
  ├── 经批准后的单条动作
  └── 证据与审计记录
```

### Skill 层负责

- 接收“查看评论”“汇总互动”“拟回复”“预览好友回访”等用户请求；
- 调用执行引擎的安全动作；
- 读取结构化输出；
- 生成自然语言摘要或回复候选；
- 真实执行前向用户展示目标与动作；
- 只有用户明确确认后才触发允许的真实操作；
- 遇到 `blocked`、风控或状态未知时停止执行并说明原因。

### 执行引擎负责

- Playwright 浏览器运行；
- 持久化登录态；
- 页面适配器与 DOM 提取；
- SQLite 数据库；
- 去重、防重复执行；
- dry-run 与 execute 隔离；
- 单条真实动作；
- 运行摘要与异常证据保存。

## 1.3 项目不做的事情

以下行为不属于本项目能力范围，开发 Agent 不得自行加入：

- 批量自动点赞；
- 对所有点赞者统一回赞；
- 自动循环刷新并无人值守执行互动；
- 自动关注、自动私信或自动发布评论；
- 通过模型自行决定真实发送内容；
- 绕过验证码、滑块、安全校验或平台限制；
- 仅凭昵称匹配就执行真实回访点赞；
- 为增长、互赞或刷量目的扩大执行范围。

## 2.2 当前主要命令与真实判断

| 命令 | 当前代码能力 | 当前产品判断 |
|---|---|---|
| `npm run auth` | 打开浏览器登录并保存 Profile | 可保留 |
| `npm run db:init` | 初始化 SQLite 数据库 | 可保留 |
| `npm run interactions:inspect` | 页面诊断采集 | 可保留并强化 |
| `npm run interactions:scan -- --type comment` | 扫描评论、入库、去重 | 初版能力，可包装为 Skill 只读动作 |
| `npm run interactions:scan -- --type like` | 扫描通知面板点赞事件 | 初版能力，需继续验证可靠性 |
| `npm run comments:plan` | 输出评论 JSON 计划文件 | 已实现旧流程，需升级为审批任务 |
| `npm run comments:reply` | 支持 dry-run/execute、审批条目、结果记录 | 初版已有，需增加 Skill 状态门 |
| `npm run likes:plan` | 已包含通知 → 主页 → 视频候选定位逻辑 | 应改为"回访候选预览" |
| `npm run likes:reciprocate` | ⚠️ 已存在真实点赞执行路径，**缺少 `FEATURE_DISABLED` 硬阻断门** | **MVP 必须默认禁用真实 execute** |
| `npm run history` | 命令入口存在 | 非当前优先级 |
| `npm run server` | 命令入口存在 | 非当前优先级 |

> **⚠️ 2026-05-29 代码审查发现**：`likes:reciprocate` 命令缺少代码层硬阻断。`src/config/defaults.mjs` 中 `likes.enabled: true`，`src/domain/result-codes.mjs` 缺少 `FEATURE_DISABLED` 错误码。即使文档提示"暂不可用"，传入 `--plan --execute` 即可触发真实点赞。此问题已在 S0.5 阶段标记为最高优先级修复项。

## 2.3 当前已经具备的安全基础

当前代码已出现以下安全基础，改造时不得删除：

- 公共参数默认 `dryRun: true`、`execute: false`；
- `maxItems` 默认值为 `1`；
- 失败或阻塞时可保留浏览器供人工检查；
- 扫描流程具备异常证据采集与人工恢复逻辑；
- 评论执行具备 `approved === true` 校验；
- 评论和点赞执行都存在防重复动作逻辑；
- 点赞执行会在点赞状态未知时阻断；
- 运行结果能够保存为 JSON 摘要。

> **注意**：以上安全基础主要依赖参数默认值和审批字段检查。**代码层不存在实验开关或 `FEATURE_DISABLED` 硬阻断门。** 任何人传入 `--plan --execute` 且计划中 `approved: true` 即可通过当前所有检查。这是 S0.5 必须修复的关键缺口。

## 2.4 当前存在的关键风险

| 风险 | 表现 | 必须采取的处理 |
|---|---|---|
| 缺少 Skill 入口 | 仓库根目录没有 `SKILL.md` | S1 阶段新增 |
| 文档与代码状态不一致 | README 称"还没有完整的 dry-run"，但 `run-context.mjs` 默认 `dryRun: true`；README 称点赞已加硬拦截，但代码无 `FEATURE_DISABLED` 门 | S0 立即修正 |
| **真实点赞无代码层阻断** | `defaults.mjs` 中 `likes.enabled: true`，`execute-reciprocal-likes.mjs` 无实验开关检查 | **S0.5 最高优先级修复** |
| Skill 无稳定机器输出 | Agent 不能依赖解析控制台中文日志 | 新增统一 `--json` 输出 |
| 审批方式不符合 Skill | 当前主要依赖人工修改 JSON 文件 | 引入动作请求和审批状态 |
| 点赞身份绑定不足 | 通知文本关系 + 昵称片段点击可能错配 | MVP 仅提供候选预览，真实点赞硬禁用 |
| 旧计划文件可绕过新主流程 | 用户可能直接编辑计划并传入执行 | 真实执行需校验数据库审批状态与安全开关 |
| 缺失关键错误码 | `result-codes.mjs` 缺少 `FEATURE_DISABLED`、`DRY_RUN_REQUIRED`、`EXECUTE_CONFIRMATION_REQUIRED`、`IDENTITY_NOT_VERIFIED` 等 | S0.5 + S3 逐步补齐 |

# 3. 硬性执行边界

本节为强制约束。无论是人工开发、Codex 修改代码，还是 Skill 调用执行引擎，都不得绕过。

## 3.1 默认只能观察与准备

所有可能触碰真实页面互动的流程，默认工作于：

```bash
--dry-run
```

### 默认允许的行为

- 打开浏览器；
- 使用本地已保存的登录态；
- 导航到目标页面；
- 扫描评论；
- 扫描点赞、回复等互动通知；
- 保存事件并去重；
- 获取用户昵称、关系标签、主页链接、目标视频候选等信息；
- 保存页面截图、文本、DOM/HTML 与错误证据；
- 定位将要操作的目标元素；
- 生成待处理摘要、候选回复或候选回访信息；
- 校验是否已完成过相同动作。

### 默认禁止的行为

- 点击评论发送按钮；
- 点击点赞按钮；
- 自动关注；
- 自动私信；
- 自动发布任何公开互动；
- 将仅定位成功的任务写成真实成功；
- 因为候选看起来合理而自动批准执行；
- 在身份、页面或状态未知时尝试点击验证。

## 3.2 真实评论回复必须经过双确认链路

评论真实回复必须满足以下完整状态链：

```text
评论事件已入库
  → 创建候选回复动作
  → 用户确认回复文本与目标评论
  → 写入 approved 审批状态
  → dry-run 定位目标评论成功
  → 用户确认真实发送
  → execute 单条执行
  → 页面验证成功
  → 写入 succeeded 与审计记录
```

任何一步缺失，都不得执行真实发送。

### 真实评论执行前必须校验

```text
eventId 存在且事件类型为 comment
actionRequest 状态为 approved 或 confirmed_execute
replyText 为非空字符串
已完成 dry-run 且定位结果仍有效
同一事件无成功 reply_comment 记录
本轮 executed 数量小于 maxItems
用户本轮明确确认发送
```

### 评论回复默认上限

```text
maxItems = 1
```

## 3.3 好友回访在 MVP 阶段仅允许预览

### MVP 允许行为

- 扫描点赞通知；
- 提取页面可见关系标签；
- 将 `friend` / `mutual` 作为候选信号；
- 尝试定位候选主页；
- 尝试定位最近非置顶作品；
- 尝试读取当前点赞状态；
- 输出候选报告；
- 对不确定项目记录 `blocked`。

### MVP 禁止行为

- 调用真实点赞点击逻辑；
- 通过普通 Skill 对话触发真实点赞；
- 允许用户仅说“都回一下”就执行；
- 因页面显示“朋友”就跳过身份核验；
- 因目标视频看起来正确就点击；
- 批量回赞。

### 真实点赞代码的处理规则

当前仓库已经存在真实点赞执行路径，因此必须在代码层增加默认禁用门，而不是只靠文档提醒。

建议配置：

```env
ENABLE_EXPERIMENTAL_RECIPROCAL_LIKE=false
```

当配置不是明确开启状态时：

```bash
npm run likes:reciprocate -- --plan <path> --execute
```

必须返回阻断结果：

```json
{
  "ok": false,
  "code": "FEATURE_DISABLED",
  "message": "真实回访点赞处于实验禁用状态，当前仅允许候选预览和 dry-run。"
}
```

## 3.4 未来实验性真实点赞的必要条件

只有以下条件全部满足，才允许进入实验验证：

```text
实验开关已明确开启
用户针对某一条候选明确审批
目标点赞事件来自已入库记录
actorProfileUrl 或稳定 actorProfileKey 已保存
身份绑定方式通过人工验证
目标视频 URL 可稳定定位
执行前二次核验主页与目标视频一致
当前点赞状态明确为未点赞
该 targetVideoUrl 无成功 like_work 记录
dry-run 已完成
本轮 maxItems = 1
```

任一条件不满足：

```text
status = blocked
不得点击
```

## 3.5 状态不确定必须停止

以下情况必须标记为 `blocked`，不得继续真实操作：

- 未确认登录状态；
- 未确认当前页面类型；
- 页面定位规则失效；
- 找不到目标评论；
- 找不到对应回复入口；
- 找不到通知面板；
- 无法判断关系标签；
- 无法稳定绑定目标主页；
- 找不到目标作品；
- 无法判断当前是否已点赞；
- 点击后无法确认操作是否成功；
- 出现验证码、滑块、登录异常、安全验证或风控提示；
- 执行结果与数据库状态不一致。

核心原则：

> **宁可漏掉一次互动，也不能误操作一个目标。**

## 3.6 防批量、防无人值守

### 当前允许

- 用户主动触发一次扫描；
- 用户主动确认一条评论回复；
- 未来实验阶段中，用户主动确认一条回访动作。

### 当前禁止

- 后台循环扫描后自动执行；
- 定时自动发送回复；
- 定时自动回赞；
- 一次审批多条真实操作；
- 将 `maxItems` 默认值调大；
- 将默认模式从 `dry-run` 改为 `execute`。

# 6. 数据模型与状态机

## 6.1 `interaction_events`：外部互动事实

`interaction_events` 用于记录被扫描到的外部事实，不代表系统要立即采取动作。

### 事件示例

```js
{
  id: 1,
  eventType: 'comment',
  actorName: '用户昵称',
  relation: 'unknown',
  myWorkTitle: '作品标题',
  commentText: '这个怎么使用？',
  eventTimeText: '05-29',
  fingerprint: '...',
  status: 'new'
}
```

```js
{
  id: 10,
  eventType: 'like',
  actorName: '好友昵称',
  relation: 'friend',
  myWorkTitle: '',
  commentText: null,
  eventTimeText: '05-29',
  fingerprint: '...',
  status: 'new'
}
```

### 事件状态建议

```text
new                 新扫描到，未形成处理候选
candidate_prepared  已生成候选动作或候选预览
processing          正在进行受控定位/执行
resolved            对应动作已完成或用户已明确处理
blocked             当前事件无法安全推进
skipped             按规则不处理
```

说明：旧代码中的 `planned`、`approved`、`succeeded` 可暂时兼容，但后续不建议把动作审批状态全部堆入事件表。

## 6.2 `action_requests`：计划执行的动作

新增 `action_requests` 表，用来表示“基于某个互动事件，准备做什么”。

### 建议字段

| 字段 | 类型/示例 | 含义 |
|---|---|---|
| `id` | INTEGER | 动作请求 ID |
| `event_id` | INTEGER | 来源事件 |
| `action_type` | `reply_comment` / `visit_candidate` / `like_work` | 动作类型 |
| `target_actor_name` | TEXT | 目标昵称 |
| `target_actor_profile_url` | TEXT | 目标主页 |
| `target_actor_key` | TEXT | 未来稳定身份标识 |
| `target_url` | TEXT | 评论目标或视频 URL |
| `target_title` | TEXT | 目标作品标题 |
| `proposed_text` | TEXT | 候选回复文本 |
| `status` | TEXT | 动作状态 |
| `risk_reason` | TEXT | 阻断或风险原因 |
| `created_at` | TEXT | 创建时间 |
| `updated_at` | TEXT | 更新时间 |

### 动作状态建议

```text
prepared             已准备候选动作
awaiting_approval    等待用户确认
approved             用户批准候选内容/目标
dry_run_ok           dry-run 已定位成功
execute_confirmed    用户在 dry-run 后再次确认真实执行
running              执行中
succeeded            页面证据确认真实成功
blocked              因风险或异常阻断
skipped              按规则跳过
cancelled            用户取消
```

## 6.3 `approval_records`：用户审批留痕

新增审批记录，避免仅依赖可被任意修改的 JSON 文件。

### 建议字段

| 字段 | 含义 |
|---|---|
| `id` | 审批记录 ID |
| `action_request_id` | 对应动作请求 |
| `approval_type` | `approve_candidate` / `confirm_execute` / `cancel` |
| `approved_payload_json` | 当时用户看到并确认的目标与文本快照 |
| `approved_at` | 审批时间 |
| `source` | `skill_conversation` / `manual_cli` |
| `note` | 可选说明 |

### 审批原则

- 用户批准的是某个明确动作快照，不是无限授权；
- 回复文本变化后，原审批失效，必须重新批准；
- 目标 URL 或目标用户变化后，原审批失效；
- 真实执行前必须存在 `confirm_execute` 类型的审批记录；
- 旧 JSON 的 `approved: true` 只能用于兼容调试，不能单独授权 Skill 真实执行。

## 6.4 `actions`：最终动作审计

每次真实执行、明确跳过或被安全门阻断，都应记录到 `actions`。

### 评论动作记录示例

```js
{
  eventId: 1,
  actionRequestId: 100,
  actionType: 'reply_comment',
  targetTitle: '作品标题',
  targetUrl: null,
  actionText: '我整理好配置步骤后发一篇详细教程。',
  status: 'succeeded',
  reason: null,
  evidenceJson: '{"dryRun":"...","confirm":"..."}',
  screenshotPath: 'data/runs/<run-id>/evidence/reply-success/after.png',
  executedAt: '2026-05-29T00:00:00.000Z'
}
```

### 实验性点赞动作记录示例

```js
{
  eventId: 10,
  actionRequestId: 200,
  actionType: 'like_work',
  targetTitle: '候选视频标题',
  targetUrl: 'https://www.douyin.com/video/...',
  actionText: null,
  status: 'blocked',
  reason: 'FEATURE_DISABLED: MVP 阶段仅允许回访预览',
  evidenceJson: '{"relation":"friend","identityVerified":false}',
  screenshotPath: null,
  executedAt: '2026-05-29T00:00:00.000Z'
}
```

# 7. 统一命令契约

Skill 不应依赖解析终端中随时可能变化的中文日志。所有由 Skill 调用的业务命令必须支持机器可读取输出。

## 7.1 公共参数

所有涉及页面读取或真实动作的命令应统一支持：

```bash
--json
--debug
--dry-run
--execute
--keep-open
--keep-open-on-error
--pause-on-error
--max-items <number>
```

### 默认配置

```js
{
  json: false,
  debug: true,
  dryRun: true,
  execute: false,
  keepOpen: false,
  keepOpenOnError: true,
  pauseOnError: true,
  maxItems: 1
}
```

### 参数冲突

以下组合必须直接拒绝：

```bash
--dry-run --execute
```

错误输出：

```json
{
  "ok": false,
  "code": "INVALID_ARGUMENTS",
  "message": "--dry-run 与 --execute 不可同时使用。"
}
```

## 7.2 机器输出格式

启用 `--json` 后，标准输出必须提供稳定 JSON；调试日志应写入 stderr 或日志文件，避免破坏 JSON 解析。

### 成功示例

```json
{
  "ok": true,
  "command": "interactions:report",
  "mode": "read-only",
  "runId": "2026-05-29_10-30-00_report-pending",
  "data": {
    "items": []
  },
  "summary": {
    "total": 0,
    "blocked": 0
  },
  "evidence": []
}
```

### 阻断示例

```json
{
  "ok": false,
  "command": "comments:execute",
  "mode": "execute",
  "code": "DRY_RUN_REQUIRED",
  "message": "真实发送前必须先完成 dry-run 定位。",
  "data": {
    "actionRequestId": 12
  },
  "evidence": []
}
```

## 7.3 目标命令集合

### 只读采集命令

| 命令 | 作用 | 是否可产生公开动作 |
|---|---|---|
| `npm run interactions:scan -- --type comment --json` | 扫描评论 | 否 |
| `npm run interactions:scan -- --type like --json` | 扫描通知点赞候选 | 否 |
| `npm run interactions:report -- --status pending --json` | 输出待处理清单 | 否 |
| `npm run visits:prepare -- --json` | 输出回访候选预览 | 否 |

### 评论协作命令

| 命令 | 作用 | 边界 |
|---|---|---|
| `npm run comments:prepare -- --event-id <id> --reply-text "<文本>" --json` | 创建候选回复动作 | 不发送 |
| `npm run actions:approve -- --action-id <id> --type approve_candidate --json` | 写入候选审批 | 不发送 |
| `npm run comments:execute -- --action-id <id> --dry-run --json` | 定位目标评论 | 不发送 |
| `npm run actions:approve -- --action-id <id> --type confirm_execute --json` | dry-run 后确认真实执行 | 不发送 |
| `npm run comments:execute -- --action-id <id> --execute --max-items 1 --json` | 真实发送一条 | 必须通过安全门 |

### 回访候选与实验命令

| 命令 | MVP 处理 |
|---|---|
| `npm run visits:prepare -- --json` | 可用，只输出候选 |
| `npm run visits:execute -- --action-id <id> --dry-run --json` | 仅开发验证使用 |
| `npm run visits:execute -- --action-id <id> --execute --max-items 1 --json` | 默认返回 `FEATURE_DISABLED` |

## 7.4 旧命令兼容策略

当前已有命令可在迁移期保留：

```bash
npm run comments:plan
npm run comments:reply -- --plan <path>
npm run likes:plan
npm run likes:reciprocate -- --plan <path>
```

但必须满足：

- README 不再将旧 JSON 流程描述为 Skill 主流程；
- 旧命令执行真实回复也必须遵循 `dry-run` 与单条限制；
- 旧点赞 `--execute` 在 MVP 阶段同样被实验禁用门阻断；
- 完成新动作请求链路后，将旧计划命令标记为 legacy/debug only。

|---|---|---|---|---|
| `SUCCESS` | 成功完成当前阶段 | - | 按阶段规则 | ✅ 已实现 |
| `INVALID_ARGUMENTS` | 参数错误或冲突 | 否 | 禁止 | ✅ 已实现（隐式） |
| `FEATURE_DISABLED` | 实验功能未开启 | 否 | 禁止 | ❌ **缺失** — S0.5 最高优先级 |
| `LOGIN_REQUIRED` | 页面需要重新登录 | 是 | 禁止 | ✅ 已实现 |
| `NAVIGATION_TIMEOUT` | 页面导航超时 | 是 | 禁止 | ✅ 已实现 |
| `WRONG_PAGE` | 当前页面不是目标页 | 是 | 禁止 | ✅ 已实现 |
| `RATE_LIMITED` | 疑似风控/安全校验 | 否 | 立即停止本轮 | ✅ 已实现 |
| `UNKNOWN_ERROR` | 未分类异常 | 人工判断 | 禁止 | ✅ 已实现 |

| 错误码 | 含义 | 状态 | 当前实现 |
|---|---|---|---|
| `NOTIFICATION_PANEL_NOT_FOUND` | 找不到互动通知面板 | `blocked` | ✅ 已实现 |
| `NOTIFICATION_ITEMS_EMPTY` | 无可识别通知 | `blocked` 或空结果 | ✅ 已实现 |
| `ACTOR_PROFILE_LINK_NOT_FOUND` | 找不到用户主页入口 | `blocked` | ✅ 已实现 |
| `RELATION_UNKNOWN` | 无法确认候选关系 | `blocked` | ✅ 已实现 |
| `IDENTITY_NOT_VERIFIED` | 未完成稳定身份绑定 | `blocked` | ❌ **缺失** — S4/S5 补齐 |
| `PROFILE_PAGE_NOT_FOUND` | 主页不可访问或无法确认 | `blocked` | ✅ 已实现 |
| `VIDEO_LIST_NOT_FOUND` | 找不到作品列表 | `blocked` | ✅ 已实现 |
| `LATEST_VIDEO_NOT_FOUND` | 无法确定目标作品 | `blocked` | ✅ 已实现 |
| `LIKE_STATE_UNKNOWN` | 无法判断点赞状态 | `blocked` | ✅ 已实现 |
| `ALREADY_LIKED` | 已点赞过 | `skipped` | ✅ 已实现 |
| `RECIPROCAL_LIKE_DISABLED` | MVP 禁止真实回赞 | `blocked` | ❌ **缺失** — S0.5 最高优先级 |

## 8.4 成功写入规则

### 评论回复

只有在页面已验证显示回复成功后，才能写：

```text
actions.status = succeeded
action_requests.status = succeeded
interaction_events.status = resolved
```

以下情况均不得写成功：

- 只是定位到了评论；
- 只是打开了输入框；
- 只是填写了文本；
- 只是点击了发送按钮；
- 页面结果无法确认。

### 回访点赞

MVP 阶段不存在公开可写成功的真实点赞路径。

未来实验验证中，只有在点击后重新读取页面状态，且可确认由未点赞变为已点赞后，才能写 `succeeded`。若状态未知，必须写 `blocked`，不得再次点击。

# 9. 运行上下文与证据保存

## 9.1 每轮运行必须有 `runId`

建议格式：

```text
2026-05-29_10-30-00_scan-comments
2026-05-29_10-32-10_execute-comment-reply
```

## 9.2 输出目录

```text
data/runs/
  <run-id>/
    summary.json
    result.json
    evidence/
      <step-name>/
        screenshot.png
        page-text.txt
        page.html
        page-info.json
        failure.json
```

## 9.3 `summary.json` 最小字段

```json
{
  "command": "comments:execute",
  "mode": "dry-run",
  "startedAt": "2026-05-29T02:30:00.000Z",
  "finishedAt": "2026-05-29T02:30:15.000Z",
  "scanned": 0,
  "planned": 0,
  "executed": 0,
  "succeeded": 0,
  "failed": 0,
  "blocked": 0,
  "skipped": 0,
  "browserKeptOpen": true,
  "evidenceDirectories": []
}
```

## 9.4 必须保存证据的场景

- 登录失效；
- 页面类型无法确认；
- 元素定位失败；
- 评论解析失败且影响处理；
- 目标评论无法定位；
- 真实执行后无法确认成功；
- 通知条目无法对应用户主页；
- 关系标签或身份不明确；
- 视频或点赞状态无法确认；
- 发生验证码或风控拦截；
- 未处理异常。

## 9.5 浏览器关闭规则

### 只读扫描成功

- 默认写摘要后关闭；
- 传入 `--keep-open` 时保留。

### 出现阻断或异常

- 必须先写结果和证据；
- 默认按 `keepOpenOnError = true` 保留浏览器；
- 允许人工选择重新检测、保存诊断、退出并关闭、退出但保持打开。

### 真实执行流程

无论执行成功或失败，顺序必须为：

```text
形成当前条目结果
→ 写动作/状态记录
→ 写本轮 result.json
→ 写 summary.json
→ 再决定关闭或保留浏览器
```

不得为了保留浏览器而阻塞结果落盘。

# 10. Skill 调用规则

## 10.1 `SKILL.md` 必须表达的内容

根目录 `SKILL.md` 至少应明确：

- 此 Skill 用于处理抖音创作者真实互动；
- 默认只扫描、汇总、生成建议与预览；
- 评论真实回复必须逐条确认且先 dry-run；
- 真实回复每轮最多一条；
- 好友回访 MVP 仅预览，禁止真实点赞；
- 状态不确定或脚本返回阻断时必须停止；
- 不向用户展示敏感登录态数据。

## 10.2 用户意图与工具动作映射

| 用户请求 | Skill 应做的事 | 是否需确认 |
|---|---|---:|
| “看看有没有新评论” | 扫描评论 + 输出待处理报告 | 否 |
| “谁给我点赞了” | 扫描通知 + 汇总候选 | 否 |
| “帮我回复这些评论” | 展示评论并生成回复草稿 | 不发送 |
| “用这句话回复第 2 条” | 创建候选动作并展示预览 | 需要后续确认 |
| “先看看能不能定位到那条评论” | 执行 dry-run | 否 |
| “确认发送这一条” | 写入最终确认并执行单条回复 | 是 |
| “好友给我的点赞，帮我回一下” | 生成回访候选并说明 MVP 不执行点赞 | 不执行 |
| “自动把这些都点赞回去” | 拒绝批量真实执行，提供候选预览 | 禁止 |

## 10.3 Skill 输出给用户的最少信息

### 评论候选展示

```text
目标用户：
对应作品：
原评论：
建议回复：
当前模式：预览 / dry-run / 待发送
需要确认的下一步：
```

### 回访候选展示

```text
候选用户：
页面识别关系：
目标主页：
候选视频：
是否已点赞：
是否可继续：仅预览 / 阻断原因
```

## 任务 06：评论单条真实执行接入审批状态机

### 目标

完成 Skill 的第一个安全可用闭环。

### 必须完成

1. 真实执行读取 action request；
2. 未审批不得执行；
3. 未 dry-run 成功不得执行；
4. 未有最终发送确认不得执行；
5. 执行前检查重复动作；
6. 成功必须页面验证；
7. 单次最多一条；
8. 返回结构化结果并保留审计证据。

### 验收

完成以下真实流程：

```text
用户查看评论
→ Skill 生成候选回复
→ 用户批准
→ dry-run 定位成功
→ 用户确认发送
→ 单条发送成功
→ 再次执行被拦截
```

# 16. 上线范围与完成标准

## 16.1 MVP 可对外描述的能力

完成 S0-S4 后，可以对外介绍为：

> 礼尚往来是一款面向抖音创作者的 OpenClaw 互动助手 Skill。它可以扫描真实评论和互动通知、整理待处理事项、生成回复建议，并在本人明确确认后安全执行单条评论回复。好友点赞回访当前提供候选预览，不默认执行真实点赞。

## 16.2 MVP 完成标准

- [ ] 根目录存在可加载的 `SKILL.md`；
- [ ] README、产品计划、改造计划和执行边界一致；
- [ ] 扫描与待办报告可以被 Skill 稳定调用；
- [ ] 评论候选回复可在对话中准备和确认；
- [ ] 评论真实发送满足审批、dry-run、最终确认、单条上限和成功验证；
- [ ] 真实回访点赞在代码层默认禁用；
- [ ] 所有真实动作均有审计与证据；
- [ ] 异常或风控场景会停止而不是继续尝试。

## 16.3 非 MVP 完成标准

以下能力不得被当作首版完成条件：

- 管理后台；
- 定时扫描；
- 多账号；
- 批量回复；
- 批量回赞；
- 公开可用的真实点赞回访。

# 17. 最终开发原则

1. **先把项目做成 Skill，再考虑扩大功能。**
2. **先让只读扫描稳定，再让评论单条执行可靠。**
3. **好友回访先做候选预览，真实点赞默认关闭。**
4. **审批必须可追溯，不能只依赖人工改 JSON。**
5. **所有执行结果必须可验证；无法验证就是阻断。**
6. **已有安全机制只能增强，不能为了赶进度删除。**
7. **产品的价值是“不漏掉值得回应的人”，不是“自动刷互动”。**

## 附录 B：核心安全断言

```text

ASSERT 每轮执行存在 summary 与必要证据
```
