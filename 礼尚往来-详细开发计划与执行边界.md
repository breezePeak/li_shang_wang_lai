> 项目仓库：`breezePeak/li_shang_wang_lai`

> 编写日期：2026-05-28

---

## 1. 项目目标

本项目用于辅助处理抖音账号互动，目标分为两条业务链路。

### 1.1 评论回复链路

当其他用户评论我的作品时：

1. 程序发现评论事件；
2. 将评论保存到本地数据库；
3. 生成待回复计划；
4. 用户填写或确认回复内容；
5. 用户审批后，程序执行单条回复；
6. 页面确认回复是否成功；
7. 保存成功、失败、跳过和现场证据；
8. 后续重复扫描时不得重复回复。

### 1.2 好友点赞回访链路

当其他用户给我的作品点赞时：

1. 程序扫描点赞事件；
2. 识别点赞用户；
3. 判断该用户是否为好友/互相关注；
4. 仅符合关系条件的用户生成回访计划；
5. 打开对方主页；
6. 定位其最近一条可处理视频；
7. 判断当前账号是否已经点过赞；
8. 用户审批后执行单条回赞；
9. 保存动作记录和页面证据；
10. 后续不得重复给同一目标作品回赞。

### 2.2 当前必须先解决的问题

#### 问题 A：找不到页面或元素时，扫描程序会关闭浏览器

当前 `src/cli/scan-interactions.mjs` 的整体流程中，无论成功还是失败，最终都会执行 `browser.close()`。

造成的结果：

- 页面结构没匹配上时，浏览器直接被关闭；
- 用户无法观察程序实际停在哪个页面；
- 用户无法手动切换页面后让程序重试；
- Agent 只能凭猜测不停改选择器；
- 后续点赞和评论功能都无法稳定调试。

**这是当前第一优先级问题。**

#### 问题 B：评论执行程序结束后永久挂起

当前 `src/cli/execute-comment-replies.mjs` 为了让用户查看浏览器，在结果落盘之前执行永久等待。

造成的结果：

- 回复结果 JSON 无法正常写出；
- 最终统计无法输出；
- 任务无法自然结束；
- 数据库状态与结果文件可能不一致。

#### 问题 C：评论“发送成功”的判断不可靠

当前行为接近于：

1. 找到输入框；
2. 填写内容；
3. 点击发送；
4. 等待两秒；
5. 直接认定成功。

此逻辑不可靠。  
**点击成功不等于回复成功。** 必须依据页面状态变化、已回复标记或回复内容展示来确认。

#### 问题 D：点赞回访核心业务尚未实现

当前 `likes:plan` 和 `likes:reciprocate` 仍然为 TODO。  
在 P0 与评论闭环完成之前，**禁止提前实现真实自动点赞动作**。

# 3. 硬性业务边界

本节是开发 Agent 的禁止越界规则，未经明确修改不得擅自突破。

## 3.1 默认只能观察，不允许执行真实动作

所有浏览器业务命令默认工作在：

```bash
--dry-run
```

### `--dry-run` 允许行为

- 打开浏览器；
- 使用已有登录态；
- 导航到目标页面；
- 扫描评论；
- 扫描点赞通知；
- 获取用户、作品、关系标签等候选信息；
- 保存截图、页面文本和 HTML；
- 定位“将要操作”的目标元素；
- 生成计划文件。

### `--dry-run` 禁止行为

- 点击评论发送按钮；
- 点击点赞按钮；
- 自动关注；
- 自动私信；
- 自动发布任何互动；
- 将任务写为真实成功。

只有用户显式使用：

```bash
--execute
```

并且任务满足审批及安全校验，才允许真实执行。

---

## 3.2 真实操作必须经过审批

真实执行前必须满足：

```js
item.approved === true
```

评论回复还必须满足：

```js
typeof item.replyText === 'string'
item.replyText.trim().length > 0
```

点赞回访还必须满足：

```js
item.relation === 'friend' || item.relation === 'mutual'
item.targetVideoUrl 可明确定位
item.alreadyLiked === false
```

否则不得真实执行。

---

## 3.3 非好友或关系不明，禁止回赞

只有以下关系可以进入回赞计划：

```text
friend
mutual
```

以下情况必须跳过或阻塞：

```text
unknown
following
follower
stranger
无法识别关系
没有关系标签
```

禁止行为：

- 仅凭昵称判断对方是好友；
- 关系不明时“先点了再说”；
- 对所有点赞用户统一回赞；
- 仅因为用户给我点赞，就自动视为好友。

---

## 3.4 找不到准确目标作品，禁止点赞

进入对方主页后，只有同时满足以下条件，才允许形成可审批任务：

- 能确认当前主页属于目标用户；
- 能提取稳定的作品卡片或视频链接；
- 能明确目标作品选择规则；
- 能判断该作品当前是否已点赞；
- 能保存目标作品的稳定唯一标识或 URL。

以下情况必须终止该条任务：

- 用户主页打不开；
- 页面用户与目标用户不一致；
- 找不到作品列表；
- 不能判断哪个是最近可处理作品；
- 不能判断是否已经点赞；
- 目标作品没有稳定标识；
- 页面结构变化导致判断不可靠。

---

## 3.5 不允许无限循环和批量猛操作

初期只支持人工触发的一次性命令，不实现后台无限轮询。

所有真实执行命令必须支持：

```bash
--max-items <number>
```

初期默认值：

| 操作 | 默认一次最大执行量 |
|---|---:|
| 评论真实回复 | 1 |
| 点赞真实执行 | 1 |
| 扫描入库 | 最多 50，允许配置 |

在经过真实账号小规模验证前，禁止将真实执行默认上限调大。

---

## 3.6 状态不确定时必须停止，不得试点

以下情况必须记为 `blocked`，不得继续点击：

- 未确认是否登录；
- 未确认当前页面类型；
- 找不到目标评论；
- 找不到目标回复按钮；
- 找不到目标用户主页；
- 找不到目标作品；
- 不知道按钮当前是否已点赞；
- 点击后无法验证是否成功；
- 出现滑块、验证码、异常提示或疑似风控。

原则：

> **宁可少执行，不可误执行。**

## 5.2 新增公共 CLI 参数

所有涉及浏览器的命令统一支持：

```bash
--debug
--dry-run
--execute
--keep-open
--keep-open-on-error
--pause-on-error
--max-items <number>
```

### 参数说明

| 参数 | 含义 |
|---|---|
| `--debug` | 输出详细日志并保存页面诊断数据 |
| `--dry-run` | 仅扫描、定位、生成计划，不执行真实动作；默认值 |
| `--execute` | 明确允许执行已审批真实动作 |
| `--keep-open` | 无论成功或失败，结束后均保留浏览器 |
| `--keep-open-on-error` | 失败或阻塞时保留浏览器 |
| `--pause-on-error` | 失败后进入人工处理菜单 |
| `--max-items N` | 单次最大真实执行数量 |

### 默认配置

```js
{
  dryRun: true,
  execute: false,
  debug: true,
  keepOpen: false,
  keepOpenOnError: true,
  pauseOnError: true,
  maxItems: 1
}
```

### 参数冲突处理

如果同时传入：

```bash
--dry-run --execute
```

程序必须拒绝启动，并输出：

```text
参数冲突：--dry-run 与 --execute 不可同时使用。
```

不得自行选择其中一个。

## 5.5 新增人工处理菜单

新增文件：

```text
src/browser/interactive-control.mjs
```

可恢复失败时，终端显示：

```text
[BLOCKED] 未找到评论列表
错误码：COMMENT_LIST_NOT_FOUND

现场证据已保存：
data/runs/<run-id>/evidence/comment-list-not-found/

浏览器已保持打开。你可以手动进入正确页面后重新检测。

请选择：
[r] 重新检测当前页面
[s] 跳过当前步骤或当前任务

[q] 安全退出并关闭浏览器
[k] 安全退出但保持浏览器打开
```

返回结构：

```js
{
  action: 'retry' | 'skip' | 'diagnose' | 'quit-close' | 'quit-keep-open'
```

### 菜单行为规则

| 选项 | 行为 |
|---|---|
| `r` | 重新执行当前失败步骤，不重新启动浏览器 |
| `s` | 当前步骤记为 `blocked` 或当前任务记为 `skipped`，不得标记成功 |
| `d` | 再次保存现场证据，不改变业务状态 |
| `q` | 先写本轮结果，再关闭浏览器退出 |
| `k` | 先写本轮结果，再结束脚本但不主动关闭浏览器 |

## 5.6 重构浏览器关闭规则

### 扫描类命令

#### 成功时

- 默认：写运行摘要后关闭浏览器；
- 传入 `--keep-open`：写摘要后保持打开。

#### 失败或阻塞时

- 必须保存证据；
- 默认启用 `--keep-open-on-error`；
- 必须保持浏览器可供人工查看；
- 启用 `--pause-on-error` 时进入人工菜单；
- 不允许一发生定位失败就执行 `browser.close()`。

### 执行类命令

无论成功或失败，执行顺序必须为：

1. 记录当前任务执行结果；
2. 写数据库动作状态；
3. 写本轮结果 JSON；
4. 写运行摘要；
5. 最后根据参数决定关闭还是保留浏览器。

不得为了“让用户看看页面”而在结果写入前永久等待。

## 5.7 修改 `src/cli/scan-interactions.mjs`

### 当前问题

总流程 `finally` 中无条件关闭浏览器。

### 改造要求

1. 引入公共运行参数和运行上下文；
2. 将“页面阻塞”与“程序不可恢复错误”区分开；
3. 页面阻塞使用结构化结果处理，不直接 throw；
4. 页面阻塞时调用 `captureEvidence()`；
5. 根据菜单选择决定重试、跳过或退出；
6. 只有在明确需要关闭时才执行 `browser.close()`。

### 伪代码参考

```js
let shouldCloseBrowser = true;
let page = null;
const run = createRunContext('scan-interactions', args);

try {
  const ctx = await createBrowserContext({ headless: false });
  browser = ctx.browser;
  page = getOrCreatePage(ctx.context);

  const result = await runScanFlow(page, run);

  if (!result.ok) {
    run.hadBlocked = true;

    await captureEvidence(page, {
      outputDir: run.outputDir,
      step: result.step,
      code: result.code,
      message: result.message,
      recoverable: result.recoverable
    });

    if (run.options.keepOpenOnError) {
      shouldCloseBrowser = false;
    }

    if (run.options.pauseOnError && result.recoverable) {
      const choice = await promptRecoveryAction(...);

      if (choice.action === 'retry') {
        // 在同一个 page 上重新执行当前步骤
      }

      if (choice.action === 'quit-close') {
        shouldCloseBrowser = true;
      }

      if (choice.action === 'quit-keep-open') {
        shouldCloseBrowser = false;
      }
    }
  }
} catch (err) {
  run.hadError = true;

  if (page) {
    await captureEvidence(page, {
      outputDir: run.outputDir,
      step: 'unhandled-error',
      code: 'UNKNOWN_ERROR',
      message: err.message
    });
  }

  shouldCloseBrowser = !run.options.keepOpenOnError;
} finally {
  await saveRunSummary(run);

  if (browser && shouldCloseBrowser && !run.options.keepOpen) {
    await browser.close();
  } else if (browser) {
    console.log('[browser] 浏览器保持打开，供人工检查。');
  }
}
```

### P0 验收场景

执行：

```bash
npm run interactions:scan -- --type comment --dry-run --debug --pause-on-error --keep-open-on-error
```

手动制造以下场景并验证：

| 场景 | 预期结果 |
|---|---|
| 当前未登录 | 报 `LOGIN_REQUIRED`，保存证据，浏览器不自动关闭 |
| 当前在错误页面 | 报 `WRONG_PAGE`，保存证据，可手动进入正确页面后按 `r` |
| 找不到评论列表 | 报 `COMMENT_LIST_NOT_FOUND`，保存证据，不关闭浏览器 |
| 页面跳转超时 | 报 `NAVIGATION_TIMEOUT`，保存证据，不关闭浏览器 |
| 正常扫描完成 | 写摘要；未传 `--keep-open` 时可正常关闭 |

# 6. 修复评论执行脚本的阻塞问题

## 6.1 修改 `src/cli/execute-comment-replies.mjs`

### 必须移除或重构的行为

不得在结果保存之前执行：

```js
await new Promise(() => {});
```

### 正确顺序

```text
读取计划
→ 校验任务
→ 打开浏览器
→ dry-run 定位或 execute 执行
→ 每条任务形成结果
→ 写 actions/interaction_events 状态
→ 写 reply-result JSON
→ 写本轮 summary JSON
→ 根据参数关闭或保留浏览器
```

---

## 6.2 执行前校验

每条回复任务必须按顺序检查：

```js
item.approved === true
item.eventId 存在
typeof item.replyText === 'string'
item.replyText.trim().length > 0
事件未被成功回复
没有历史 succeeded 回复动作
当前运行数量未超过 maxItems
```

### 校验不通过的处理

| 情况 | 结果码 | 状态 | 是否打开页面发送 |
|---|---|---|---|
| 未审批 | `ACTION_NOT_APPROVED` | `skipped` | 否 |
| 回复为空 | `EMPTY_REPLY_TEXT` | `blocked` | 否 |
| 已成功执行过 | `DUPLICATE_ACTION` | `skipped` | 否 |
| 超过本轮数量限制 | `MAX_ITEMS_REACHED` | `skipped` | 否 |

## 6.3 Dry-run 与 execute 分离

### Dry-run 命令

```bash
npm run comments:reply -- --plan data/plans/comments-plan.json --dry-run --max-items 1 --keep-open
```

允许：

- 打开评论页；
- 找到目标评论；
- 找到对应回复按钮；
- 截图并输出目标信息。

禁止：

- 填写回复框；
- 点击发送；
- 写入 replied/succeeded 状态。

### Execute 命令

```bash
npm run comments:reply -- --plan data/plans/comments-plan.json --execute --max-items 1 --keep-open-on-error --pause-on-error
```

仅执行审批通过且校验完整的单条任务。

| `MAX_ITEMS_REACHED` | 超过本轮执行数量 | 否 |

## 7.3 事件与动作状态

### `interaction_events.status`

```text

approved     已人工审批
running      正在执行
succeeded    页面确认执行成功
failed       已尝试且明确失败
blocked      当前信息不足或页面异常，不可安全继续
```

### `actions.status`

```text

running
succeeded
failed
blocked
skipped
```

### 状态写入规则

- 只点击按钮，不得写 `succeeded`；
- 无法确认真实成功，写 `blocked`；
- 非好友点赞事件，写 `skipped`；
- 已成功处理过的任务，写 `skipped`；
- 遇到验证码/风控，写 `blocked` 并立即停止本轮。

# 8. P1：评论扫描稳定化

## 8.1 目标

在不发送任何回复的情况下，稳定完成：

```text
打开页面 → 确认页面状态 → 提取评论 → 入库 → 去重 → 输出证据和摘要
```

## 8.2 页面确认规则

`ensureCommentPageReady()` 不得只凭 URL 判断页面准备完成。

必须确认至少一种页面特征：

- 存在评论管理标题；
- 存在评论筛选区域；
- 存在评论列表；
- 存在明确“暂无评论”状态。

返回状态：

```js
{
  ok: true,
  code: 'SUCCESS',
  data: {
    pageState: 'comments-visible' | 'empty-comments'
  }
}
```

若 URL 看似正确，但页面没有任何可验证特征：

```js
{
  ok: false,
  code: 'WRONG_PAGE',
  recoverable: true
}
```

---

## 8.3 评论提取数据结构

每条评论至少提取：

```js
{
  actorName: string,
  commentText: string,
  timeText: string,
  myWorkTitle: string,
  hasReplied: boolean,
  source: 'creator-comment-page',
  rawText: string
}
```

### 解析失败处理

如果无法稳定提取 `actorName` 或 `commentText`：

- 不得将该条写入正式互动事件；
- 保存原始文本片段用于调试；
- 将其计入 `parseFailedCount`；
- 不得猜测用户名或评论正文。

---

## 8.4 评论去重规则

评论指纹至少包含：

```text
platform + eventType + actorName + myWorkTitle + commentText + timeText
```

必须验证：

- 同一评论重复扫描不会重复入库；
- 已回复事件不会重新生成回复计划；
- 不因页面展示顺序变化而生成重复事件。

---

## 8.5 评论扫描命令

```bash
npm run interactions:scan -- --type comment --dry-run --debug --pause-on-error
```

输出摘要必须包含：

```text
提取评论总数
新增入库数量
重复跳过数量
解析失败数量
阻塞步骤
证据输出目录
```

---

## 8.6 P1 验收标准

| 场景 | 预期结果 |
|---|---|
| 正常有评论 | 正确提取并新增入库 |
| 无评论 | 正确识别空列表，不报错退出 |
| 未登录 | 保存证据，浏览器保留，可人工登录并重试 |
| 错误页面 | 保存证据，可人工切页后重试 |
| 重复扫描 | 不重复新增事件 |

此阶段禁止真实发送回复。

# 9. P2：评论回复计划与单条执行闭环

## 9.1 目标

跑通完整流程：

```text
评论扫描 → 计划生成 → 用户填写回复 → 用户审批 → dry-run 定位 → execute 单条发送 → 页面确认 → 保存动作记录 → 重复执行拦截
```

---

## 9.2 计划文件格式

```json
{
  "planType": "comment_reply",
  "mode": "manual",
  "createdAt": "2026-05-28T00:00:00.000Z",
  "items": [
    {
      "eventId": 1,
      "actorName": "用户昵称",
      "workTitle": "作品标题",
      "commentText": "内容不错",
      "commentTime": "05-28",
      "replyText": "感谢支持！",
      "approved": false,
      "status": "planned",
      "reason": ""
    }
  ]
}
```

### 计划规则

- 新计划中 `approved` 必须默认 `false`；
- `replyText` 初始可为空，由用户填写；
- 真实执行时空回复必须拦截；
- 不允许自动批准所有任务；
- 初期一次真实执行最多 1 条。

---

## 9.3 回复动作拆分

`sendReply()` 不得继续承担所有工作。建议拆分为：

```js
locateTargetComment(page, item)
openReplyBox(page, target)
fillReplyText(page, text)
clickSendReply(page)
confirmReplySucceeded(page, item)
```

每一步返回统一结构化结果。

---

## 9.4 成功确认规则

点击发送之后，至少需要满足以下一项经过真实验证的条件：

- 当前评论条目显示用户刚刚发送的回复文本；
- 当前评论条目出现“已回复”状态；
- 页面出现明确成功反馈，随后重新扫描该条确认已回复；
- 其他有证据支持的稳定成功信号。

无法确认时：

```text
code = COMMENT_SEND_UNCONFIRMED
status = blocked
```

不得写为 `succeeded`。

---

## 9.5 动作审计

评论回复执行必须写入 `actions` 表，而不是仅更新事件状态。

动作记录至少包括：

```js
{
  eventId: 1,
  planId: 2,
  actionType: 'reply_comment',
  targetTitle: '作品标题',
  actionText: '感谢支持！',
  status: 'succeeded',
  reason: null,
  evidenceJson: '{...}',
  screenshotPath: 'data/runs/.../after-send.png',
  executedAt: '...'
}
```

---

## 9.6 P2 验收标准

| 测试场景 | 要求 |
|---|---|
| `approved: false` | 不执行发送 |
| `approved: true` 且回复为空 | 拦截，错误码 `EMPTY_REPLY_TEXT` |
| dry-run | 仅定位目标评论并截图 |
| execute 单条正常回复 | 页面确认后才写成功 |
| 二次执行同一任务 | 以 `DUPLICATE_ACTION` 跳过 |
| 找不到对应评论 | 保存证据，不得回复其他条目 |
| 点击后无法确认 | 写 `blocked`，不得写成功 |
| 执行完成 | 总能生成结果 JSON 和 summary JSON |

# 10. P3：验证点赞通知是否可用于回访

## 10.1 阶段目标

本阶段只验证能力，不做真实点赞。

必须回答三个问题：

1. 网页端能否稳定获取“谁赞了我的作品”？
2. 能否稳定判断该用户是好友/互相关注？
3. 能否稳定获取或进入该用户主页？

三项中任意一项不能稳定实现，则不能进入真实回赞开发。

## 10.2 点赞事件数据结构

```js

  eventType: 'like',
  actorName: string,
  actorProfileUrl: string | null,
  actorProfileKey: string | null,
  relation: 'friend' | 'mutual' | 'following' | 'follower' | 'unknown',
  likedTargetType: 'work' | 'comment' | 'unknown',
  myWorkTitle: string | null,
  timeText: string,
  source: 'notification-panel',
  rawText: string
}
```

## 10.3 点赞通知边界处理

| 场景 | 处理方式 |
|---|---|

| 显示朋友/互相关注且主页入口明确 | 可进入候选计划 |
| 对方赞的是评论，不是作品 | 初期跳过 |
| 通知列表无法完整解析 | 保存证据，不得猜测 |

## 10.4 页面验证报告

```md
- 关系标签是否稳定可见
- 用户主页入口如何获取
- 使用的稳定特征
- 不稳定特征及失败案例
- 是否允许进入 P4
```

只有报告明确写出：

```text
点赞用户身份、关系、主页入口均能稳定提取。
```

才能开发 P4。

# 15. Agent 开发行为约束

## 15.1 一次只做一个任务

Agent 每次只能完成一个可验证目标。

禁止一次任务同时做：

- 浏览器生命周期大重构；
- 评论真实发送；
- 点赞真实执行；
- 本地管理页面；
- 数据库架构扩展；
- 后台定时任务。

正确做法：

# 16. 可直接逐条交给 Agent 的任务单

## 任务 01：失败时保留浏览器并保存证据

### 任务目标

解决：程序找不到页面或元素时立即关闭浏览器，无法调试的问题。

### 允许修改内容

- 新增公共运行参数解析；
- 新增 `src/browser/run-context.mjs`；
- 新增 `src/browser/failure-evidence.mjs`；
- 新增 `src/browser/interactive-control.mjs`；
- 修改 `src/cli/scan-interactions.mjs`；
- 必要时小幅调整页面诊断模块。

### 必须实现

1. 默认运行模式为 `--dry-run`；
2. 支持 `--debug`、`--keep-open`、`--keep-open-on-error`、`--pause-on-error`；
3. 页面定位失败时保存截图、页面文本、HTML、`failure.json`；
4. 失败后浏览器保持打开；
5. 人工可选择重新检测当前页面；
6. 用户明确退出后才按选择关闭或保留浏览器；
7. 每轮运行写出 `summary.json`。

### 禁止修改范围

- 不实现点赞计划；
- 不实现点赞执行；
- 不执行真实评论发送；
- 不实现管理后台；
- 不删除现有页面采集能力。

### 验收命令

```bash
npm run interactions:scan -- --type comment --dry-run --debug --pause-on-error --keep-open-on-error
```

### 验收结果

当程序位于错误页面或未找到评论列表时：

- 终端输出明确错误码；
- 生成现场证据目录；
- 浏览器没有自动关闭；
- 用户可手动进入正确页面；
- 选择 `r` 后程序能重新检测。

## 任务 02：修复评论执行结果不能正常落盘

### 任务目标
