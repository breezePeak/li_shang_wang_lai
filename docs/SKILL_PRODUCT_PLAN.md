> 文档用途：定义项目转为 OpenClaw Skill 后的产品定位、能力边界、MVP 范围与开发路线。  
> 建议存放路径：`docs/SKILL_PRODUCT_PLAN.md`  
> 文档状态：规划稿 v1.0  
> 项目名称：礼尚往来（li_shang_wang_lai）

---

## 1. 背景与定位调整

当前项目已经具备一套基于 Playwright + SQLite 的本地执行能力，包括：

- 复用抖音登录态；
- 扫描评论与点赞/互动通知；
- 将互动事件保存到本地数据库并去重；
- 生成评论回复计划；
- 执行评论回复的 dry-run 与单条真实操作；
- 生成点赞回访计划；
- 保存执行过程和失败证据。

但现阶段项目的使用方式仍以 CLI 命令和手工编辑 JSON 计划为主。这适合作为执行引擎，不适合作为最终产品体验。

本次定位调整的核心结论是：

> **“礼尚往来”应定位为创作者互动助手 Skill，而不是自动互赞或自动运营脚本。**

Skill 负责理解用户意图、组织流程、展示候选、请求审批；现有 Node.js 代码负责浏览器操作、页面采集、本地记录和经批准后的最小动作执行。

## 2. 产品定位

### 2.1 产品名称

**礼尚往来 · 创作者互动助手 Skill**

### 2.2 一句话介绍

帮助抖音创作者发现新评论和好友互动、生成回复或回访建议，并在本人明确确认后安全执行单条互动。

### 2.3 目标用户

- 希望及时回复真实评论的个人创作者；
- 作品有一定互动量、容易漏掉评论和好友点赞的人；
- 愿意让 AI 帮助整理信息和拟回复，但不愿把账号交给自动脚本的人；
- 需要操作留痕、希望避免重复回复或重复回访的人。

### 2.4 核心价值

项目不是追求“更多自动互动”，而是解决三个实际问题：

1. **看见**：及时发现新增评论、点赞及待处理事项；
2. **判断**：由 Skill 帮用户归类、汇总、拟回复、生成回访候选；
3. **执行可控**：真实评论回复或回访动作必须经过用户逐条确认，并保留记录。

---

## 3. 产品边界

### 3.1 要做的能力

- 扫描创作者页面中的新评论；
- 扫描互动通知中的点赞、评论或回复事件；
- 对事件入库并进行去重；
- 通过对话输出待处理摘要；
- 根据评论内容生成自然回复建议；
- 对明确确认的一条评论执行 dry-run 或真实回复；
- 将好友/互关点赞整理成“回访候选”；
- 在身份核验链路稳定后，再支持明确确认后的单条回访点赞；
- 保存扫描、计划、执行、失败与阻断记录。

### 3.2 明确不做的能力

- 不做批量自动点赞；
- 不做刷互动、互赞涨粉工具；
- 不做无人值守的循环真实操作；
- 不做自动随机评论或批量回复；
- 不绕过验证码、登录校验或平台风控；
- 不允许 Agent 在没有用户明确批准时发送公开互动；
- 不允许仅凭昵称相同就执行回访点赞。

## 4. Skill 与执行引擎的职责划分

### 4.1 Skill 层负责什么

Skill 面向用户，负责：

- 识别“查看评论”“汇总互动”“拟回复”“预览回访候选”等意图；
- 调用本地脚本完成只读扫描；
- 将结构化结果转为清晰的自然语言摘要；
- 根据用户要求生成回复候选；
- 在任何真实动作发生前展示目标、内容与风险；
- 等待用户明确确认；
- 只调用单条、已审批、受限制的执行动作；
- 将执行结果反馈给用户。

### 4.2 Node.js 执行引擎负责什么

现有代码继续负责：

- Playwright 浏览器与持久化登录态；
- 页面定位、互动提取和 DOM 适配；
- SQLite 数据存储、事件去重和状态流转；
- dry-run 目标定位；
- 单条评论回复动作；
- 后续经验证后的单条点赞动作；
- 运行摘要、截图和异常证据。

### 4.3 推荐结构

```text
OpenClaw Agent
  └── 礼尚往来 Skill（SKILL.md）
        ├── 扫描并汇总互动
        ├── 生成回复建议
        ├── 请求用户审批
        └── 调用受控动作
              ↓
Node.js 执行引擎
  ├── Playwright
  ├── SQLite
  ├── 页面 Adapter
  ├── dry-run
  ├── 单条执行
  └── 操作留痕
```

## 5. 用户体验流程

### 5.1 查看新评论

用户输入：

```text
看看我最近有没有新评论。
```

Skill 行为：

1. 调用评论扫描；
2. 保存新增评论并去重；
3. 查询待处理评论；
4. 返回摘要，例如：

```text
发现 3 条新的待处理评论：

1. 张三，在《AI 编程工具实测》下评论：这个怎么配置？
2. 小北，在《Cesium 特效展示》下评论：效果挺酷的。
3. 阿文，在《Claude Code 使用技巧》下评论：有教程吗？

需要我为其中某几条拟回复吗？
```

此过程不得发送任何回复。

### 5.2 评论回复协作

用户输入：

```text
给第 1 条回复：我整理好配置步骤后发一篇详细教程。
```

Skill 行为：

1. 展示准备回复的评论对象、原评论、回复文本；
2. 生成审批任务；
3. 执行 dry-run，确认能定位到目标评论；
4. 返回预览结果；
5. 用户再次明确说“发送”后，只执行这一条回复；
6. 保存执行结果并告知用户。

### 5.3 好友回访候选

用户输入：

```text
看看最近有哪些朋友给我点赞了，列一下可以回访的作品。
```

Skill 行为：

1. 扫描点赞通知；
2. 识别页面中标记为“朋友”或“互相关注”的候选；
3. 对每名候选定位主页及最近一条非置顶作品；
4. 检查目标作品是否已经点赞；
5. 返回候选清单；
6. MVP 阶段停止在预览，不执行真实点赞。

## 6. 安全策略

### 6.1 总体规则

| 规则 | 具体约束 |
|---|---|
| 默认只读 | 默认仅允许扫描、汇总、生成候选和 dry-run |
| 明确审批 | 真实动作必须由用户针对具体条目明确确认 |
| 单条执行 | MVP 真实执行默认最多 1 条 |
| 先预览后执行 | 在执行前展示账号对象、目标内容和动作 |
| 状态未知即阻断 | 页面定位、关系判断或点赞状态不确定时不得继续 |
| 防重复 | 已成功执行过的事件或目标不得重复操作 |
| 可追溯 | 保存计划、执行结果、运行摘要和异常证据 |
| 风控停止 | 遇到验证码、登录失效、页面异常时立刻停止 |

### 6.2 评论回复安全门

评论回复只有满足以下条件才能执行：

- 评论事件已入库且能够唯一定位；
- 用户已经看到回复内容；
- 用户明确批准该条回复；
- dry-run 成功定位目标评论；
- 单轮执行上限未超过 1；
- 该评论此前不存在成功回复记录。

### 6.3 点赞回访安全门

真实回访点赞在 MVP 阶段默认关闭。未来要开放，必须完成：

- 保存并验证目标用户的稳定身份标识，不能只靠昵称；
- 保存并重新核验目标主页 URL 与视频 URL；
- 能可靠判断目标作品是否已经点赞；
- 用户可见并确认“哪个人、哪条作品、为何选择它”；
- 真实执行受实验开关控制，默认每轮最多 1 条；
- 完成足够多的小规模人工验证。

## 7. MVP 能力清单

### 7.1 MVP 必须完成

| 模块 | 说明 | 优先级 |
|---|---|---|
| Skill 根入口 | 增加根目录 `SKILL.md`，支持 OpenClaw 安装与调用 | P0 |
| 产品文档 | README 与 docs 统一产品定位、安全边界和功能状态 | P0 |
| 评论扫描 | 获取评论、入库与去重 | P0 |
| 通知扫描 | 获取点赞/评论通知并结构化入库 | P0 |
| 待办汇总 | 提供给 Skill 读取的结构化 pending report | P0 |
| 评论回复建议 | Skill 根据评论生成候选文本 | P0 |
| 评论 dry-run | 定位目标评论但不发送 | P0 |
| 单条评论回复 | 明确确认后执行一条回复 | P1 |
| 好友回访候选 | 提取候选主页与视频，仅预览 | P1 |
| 操作审计 | 状态、执行结果、异常证据可查询 | P0 |

### 7.2 暂不纳入 MVP

| 能力 | 原因 |
|---|---|
| 真实好友回赞 | 身份绑定与目标校验尚未达到可公开执行水平 |
| 本地管理后台 | 先验证 Skill 对话闭环是否有价值 |
| 定时无人值守运行 | 存在真实账号互动风险 |
| 批量互动执行 | 与“人工确认、单条可控”定位冲突 |
| 多账号运营 | 会显著增加风险与工程复杂度 |

### 阶段 1：建立 Skill 基础骨架

#### 任务

- 根目录新增 `SKILL.md`；
- 定义 Skill 可触发的意图和流程；
- 定义真实动作必须审批的规则；
- 为执行命令提供稳定的 JSON 输出；
- 完成 OpenClaw 本地安装与调用测试。

#### 验收标准

用户在 OpenClaw 中输入：

```text
看看我有没有新的评论。
```

Skill 能调用脚本、完成扫描并返回待处理摘要，而无需用户手工运行命令。

---

### 阶段 2：完成只读互动收件箱

#### 任务

- 统一评论与通知事件的数据模型；
- 增加 pending report 命令；
- 支持按类型、状态、时间查询；
- 在解析失败时保存诊断信息并返回阻断原因；
- 优化重复事件识别。

#### 验收标准

Skill 可稳定回答：

- 有多少条新评论；
- 谁点赞了作品；
- 哪些互动尚未处理；
- 哪些事件解析失败或需要人工检查。

---

### 阶段 3：完成评论回复闭环

#### 任务

- 增加“审批任务”数据结构；
- 由 Skill 生成与呈现回复候选；
- 用审批命令代替手改 JSON；
- dry-run 成功后才允许进入真实发送；
- 每次最多执行一条；
- 向 Skill 返回结构化执行结果。

#### 验收标准

用户可以在对话中完成：

```text
生成回复建议 → 确认某一条 → dry-run → 确认发送 → 得到结果
```

---

### 阶段 4：好友回访候选预览

#### 任务

- 优化关系文本识别；
- 收集目标主页与目标作品证据；
- 对不确定身份标记为 blocked；
- 输出回访候选报告；
- 继续禁止默认真实点赞。

#### 验收标准

Skill 能返回一份可核对的回访清单，但不在未批准情况下执行任何点赞。

---

### 阶段 5：实验性单条回访点赞

#### 前置条件

- 目标身份绑定稳定；
- 目标视频核验可重复；
- 状态判断可靠；
- 审批链路完备；
- 已通过人工回归测试。

#### 任务

- 增加实验功能开关；
- 只允许逐条确认；
- 真实执行每轮最多一条；
- 执行前后保存证据；
- 失败立即阻断。

#### 验收标准

在小范围验证中，不发生目标错配、重复点赞或未知状态点击。

---

## 9. 成功衡量标准

项目的成功不以“自动做了多少互动”为指标，而应以以下结果评估：

- 用户能否更快发现漏掉的真实评论；
- 评论回复是否能被准确定位并受控发送；
- 执行记录是否完整、可检查；
- 页面异常时是否能停止而不是误操作；
- 用户是否明显减少重复翻看通知和手工整理时间；
- 项目是否始终保持“创作者协作助手”的定位。

## 10. 对外介绍文案

> **礼尚往来**是一款面向抖音创作者的 OpenClaw 互动助手 Skill。它帮助你发现新评论与好友互动、整理待处理事项、生成回复建议，并在你明确确认后安全执行单条互动。默认只预览、不自动操作，所有真实回复与点赞均遵循人工审批与操作留痕原则。
