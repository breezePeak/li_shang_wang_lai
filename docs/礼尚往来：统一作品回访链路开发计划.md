# 第一阶段：通知中心统一采集互动事件，并保存用户主页地址

## 1. 本阶段只解决什么问题

第一阶段只负责建立一份可靠的互动事件数据源：

```text
打开通知中心
→ 获取评论和点赞数据
→ 同时获取每个互动用户的主页地址
→ 与数据库已有事件比对去重
→ 新事件入库，已有事件补充缺失信息
→ 输出采集报告
```

本阶段不做：

* 回复评论；
* 生成回复内容；
* 生成好友回访计划；
* 进入用户主页查找作品；
* 检查对方作品是否已点赞；
* 点赞或发表评论。

## 2. 页面职责必须分开

### 2.1 通知中心：唯一的互动发现入口

所有新互动都先从通知中心获取：

```text
通知中心
  ├── comment：谁评论了我的作品、评论内容、关系、用户主页地址
  └── like：谁点赞了我的作品、关系、用户主页地址
```

通知中心采集到的数据，是后续所有计划的基础。

### 2.2 评论回复管理页面：只用于执行回复

后续需要回复某条评论时：

```text
从数据库读取 comment 事件
→ 进入评论回复管理页面
→ 定位该条原评论
→ dry-run
→ 审批后执行回复
```

评论回复管理页面不再承担“发现新评论并建立事件池”的职责。

### 2.3 好友主页：只用于后续回访

后续需要回访好友作品时：

```text
从数据库读取已保存的 actorProfileUrl
→ 进入好友主页
→ 找最近作品
→ 检查是否已点赞
```

第一阶段不得为了排回访计划而进入好友主页。

## 3. 统一采集命令

改造现有扫描命令或新增同步命令，推荐最终保留下面这一种主入口：

```bash
npm run interactions:scan -- --source notification --type all --json
npm run interactions:scan -- --source notification --type comment --json
npm run interactions:scan -- --source notification --type like --json
```

### `--type all`

```text
只打开通知中心
→ 同时提取 comment 和 like
→ 去重后入库
```

### `--type comment`

```text
只打开通知中心
→ 仅保留 comment 事件
→ 去重后入库
```

### `--type like`

```text
只打开通知中心
→ 仅保留 like 事件
→ 去重后入库
```

禁止行为：

```text
--type comment 时进入评论回复管理页面采集新评论
--type all 时先去评论回复管理页面采评论、再去通知中心采评论
```

原因：评论只能有一个事件来源，否则会造成重复入库和后续动作绑定混乱。

## 4. 采集事件时必须获取主页地址

通知中心解析每条评论或点赞时，必须同时尽可能拿到：

```text
actorProfileUrl
actorProfileKey
```

其中：

```text
actorProfileUrl：后续进入该用户主页进行回访的入口
actorProfileKey：用于识别同一个用户、辅助去重和合并回访来源
```

### 4.1 优先方式：直接从通知 DOM 获取链接

优先从通知节点中的头像、昵称或用户链接元素提取：

```text
href
data-user-id
sec_uid / user_id / profile key
```

这种方式只读，不改变页面状态，优先使用。

### 4.2 兜底方式：无法从 DOM 提取时定位主页地址

如果评论或点赞通知可点击头像进入主页，但通知节点不暴露链接，则允许在采集阶段：

```text
记录当前通知项唯一信息
→ 点击对应通知用户头像或昵称
→ 获取跳转后的主页 URL
→ 返回通知中心继续扫描
```

但必须满足：

* 只为补全用户主页地址，不访问该用户作品；
* 点击前必须能精确定位当前通知项，不能仅按昵称模糊点击；
* 跳转后必须返回通知中心并继续处理；
* 无法确定跳转用户与通知用户一致时，该事件标记为 `profile_unresolved`，不得猜测主页。

## 5. 每条事件保存字段

通知中心采集出的原始互动事件至少保存：

```json
{
  "eventType": "comment | like",
  "notificationItemKey": "通知项稳定标识，如可获取",
  "platformEventId": "互动事件稳定ID，如可获取",
  "actorName": "互动用户昵称",
  "actorProfileKey": "用户稳定标识，如可获取",
  "actorProfileUrl": "用户主页地址",
  "relation": "friend | mutual | follower | unknown",
  "myWorkTitle": "被互动作品标题，如可获取",
  "myWorkUrl": "被互动作品地址，如可获取",
  "commentText": "评论文字；like 时为 null",
  "eventTimeText": "通知显示时间",
  "rawPayloadJson": {
    "rawText": "通知原始文本",
    "extractSource": "notification",
    "profileResolveMethod": "dom_href | click_resolve | unresolved"
  },
  "status": "new"
}
```

关键约束：

* `comment` 与 `like` 都要保存 `actorProfileUrl`；
* `commentText` 只有评论事件必须保存；
* 第一阶段即使拿不到 `myWorkTitle`，也不能丢弃互动事件；
* 第一阶段如果拿不到 `actorProfileUrl`，事件仍可入库，但必须标记不能进入后续回访计划。

## 6. 采集时的数据库去重逻辑

每提取一条通知，必须在入库前与数据库进行比对。

### 6.1 去重优先级

优先级如下：

```text
platformEventId
→ notificationItemKey
→ eventType + actorProfileKey + 互动内容 + 时间标识
→ eventType + actorProfileUrl + 互动内容 + 时间标识
→ actorName 只能作为弱兜底，不能单独判重
```

其中“互动内容”为：

```text
comment：commentText
like：被点赞动作描述 / 被点赞作品标识，如可获取
```

### 6.2 数据库已存在且信息完整

```text
找到相同事件
且已有 actorProfileUrl、relation 等字段完整
→ 不重复插入
→ 计入 duplicates
```

### 6.3 数据库已存在但字段缺失

例如第一次扫描只拿到昵称，第二次拿到了主页地址：

```text
找到相同事件
但旧记录 actorProfileUrl 为空，新记录有值
→ 更新旧记录，补充 actorProfileUrl / actorProfileKey / relation / rawPayloadJson
→ 不新建事件
→ 计入 enriched
```

### 6.4 数据库不存在

```text
没有匹配事件
→ 新建 interaction_event
→ 计入 inserted
```

### 6.5 不允许在采集阶段合并 comment 与 like

同一个好友可能同时：

```text
评论了我的作品
点赞了我的作品
```

采集阶段必须保存为两条独立事件：

```json
[
  {
    "eventType": "comment",
    "actorProfileUrl": "同一个主页"
  },
  {
    "eventType": "like",
    "actorProfileUrl": "同一个主页"
  }
]
```

两者合并成一次好友回访，是后续计划阶段做的事。

## 7. 用户主页地址缺失时的处理

主页地址是后续回访的必要字段，但不是第一阶段保存互动事件的绝对前提。

处理规则：

| 情况                            |          事件是否入库 |   是否可进入后续回访计划 |
| ----------------------------- | --------------: | ------------: |
| 有 `actorProfileUrl`，关系为好友/互关  |               是 |         后续可判断 |
| 没有 URL，但有稳定 `actorProfileKey` |               是 | 先阻断回访，等待补 URL |
| 只有昵称                          |               是 |        不能自动回访 |
| 主页跳转身份无法确认                    | 是，标记 unresolved |        不能自动回访 |

保存状态建议补充：

```json
{
  "profileResolutionStatus": "resolved | unresolved | ambiguous"
}
```

## 8. 通知列表扫描范围

通知中心可能需要滚动加载，不能只取当前首屏。

支持参数：

```bash
npm run interactions:scan -- \
  --source notification \
  --type all \
  --max-items 100 \
  --max-scrolls 20 \
  --json
```

采集逻辑：

```text
打开通知中心
→ 提取当前可见通知
→ 每条立即做数据库判重 / 补全 / 入库
→ 向下滚动加载更多
→ 再次提取并判重
→ 连续滚动没有新事件或达到上限后停止
```

停止条件：

* 达到 `--max-items`；
* 达到 `--max-scrolls`；
* 连续两轮没有发现新的通知节点；
* 页面出现异常或登录失效，立即停止并报告。

## 11. 第一阶段测试要求

### 11.1 通知采集

* 能从同一通知列表中提取 `comment` 和 `like`；
* 评论事件保存 `commentText`；
* 点赞事件 `commentText = null`；
* comment 和 like 均尝试提取 `actorProfileUrl`；
* 能保存 `actorProfileKey` 与 `relation`；
* DOM 可直接拿到主页链接时，不应发生多余跳转；
* 只能通过点击解析主页时，应能精确回到通知中心继续采集。

### 11.2 数据库判重与补全

* 相同评论通知重复扫描，不重复入库；
* 相同点赞通知重复扫描，不重复入库；
* 同一好友的一条评论和一条点赞分别保存；
* 同一好友的不同评论分别保存；
* 旧事件缺少 `actorProfileUrl`，新扫描拿到 URL 时更新原事件，不新增重复记录；
* 不能只凭昵称将事件判定为重复。

### 11.3 命令流程

* `--source notification --type all` 不进入评论回复管理页面采集事件；
* `--source notification --type comment` 只保存通知中的评论；
* `--source notification --type like` 只保存通知中的点赞；
* 扫描阶段不得创建 `reply_comment`；
* 扫描阶段不得创建 `visit_work`；
* 扫描阶段不得进入好友作品页；
* 扫描阶段不得发生任何真实回复、点赞、评论。

## 12. 第一阶段验收标准

真实页面验证时必须看到：

```text
1. 通知中心中的评论和点赞都能被读取；
2. 每条互动尽量拿到正确的 actorProfileUrl；
3. 同一好友的 comment 和 like 均独立入库；
4. 重复执行扫描不会重复增加相同事件；
5. 第一次缺主页地址、后一次能解析到时，原记录被补全；
6. 整个阶段没有生成回复/回访计划，也没有进入对方作品执行互动。
```

满足这些条件后，才进入第二阶段：基于事件池拆分“评论回复计划”和“好友回访计划”。
